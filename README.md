# meteo-data-process-client

基于 Vue 3、Vite、Pinia 与 Element Plus 的气象数据分析与预测客户端，面向气象站点数据的查询、同步、清洗、相关性分析和模型预测展示场景。项目通过统一的接口封装、路由鉴权和业务组合式逻辑，提供从数据接入到分析结果呈现的完整前端交互闭环。

## 项目简介

本项目是气象数据分析与预测系统的前端客户端，主要服务于以下业务目标：

- 为登录用户提供受保护的气象数据查询与分析界面
- 支持按小时、按天、按时间段、按复合条件检索气象数据
- 支持查看站点数据采集概览与月度统计信息
- 支持对多种气象要素进行相关性分析并以图表呈现
- 支持按模型类型发起短期和长期预测，并展示预测结果与模型报告
- 支持前端发起站点同步、日期范围同步、气象数据同步与清洗任务

## 核心功能概览

- 登录鉴权：通过登录/注册弹窗完成身份认证，登录态保存在 `sessionStorage`
- 受保护路由：未登录用户不能访问 `/main`、`/dataAnalyze`、`/modelPrediction`
- 多粒度查询：支持按小时、按天、按时间范围、按复合条件查询气象数据
- 数据可视化：使用 ECharts 展示日内曲线与相关性热力图
- 数据同步与清洗：支持连接同步服务、同步站点目录、同步数据并执行清洗
- 模型预测：支持短期 24 小时预测、长期多日预测及模型报告展示
- 工程升级：当前已迁移到 Vue 3 + Vite + Pinia，并补充 Vitest 单元测试

## 技术栈与工程选型

- 框架：Vue 3
- 构建工具：Vite 5
- 状态管理：Pinia
- 路由：Vue Router 4
- UI 组件：Element Plus
- HTTP 客户端：Axios
- 图表：ECharts 5
- 测试：Vitest、@vue/test-utils、jsdom
- 代码规范：ESLint

工程选型上的重点如下：

- 使用 `composables` 承载查询、预测、同步等业务流程，降低页面组件复杂度
- 使用 `service` 和 `api` 分层隔离接口调用细节与业务结果整形逻辑
- 使用 `lib/http/api-client.js` 统一管理基础地址、超时、鉴权头和重试机制
- 使用 Pinia 替代旧的 Vuex 风格目录，减少样板代码并提升状态组织清晰度
- 使用 Vite 的按需构建与分包能力，对 Vue、Element Plus、Axios、ECharts 做手动拆包

## 核心架构说明

### 路由守卫

- 路由定义位于 `src/router/routes.js`
- 受保护页面包括：
  - `/main`
  - `/dataAnalyze`
  - `/modelPrediction`
- `src/router/index.js` 在全局前置守卫中读取 Pinia 鉴权状态
- `src/router/guards.js` 负责纯函数化判定逻辑，便于单元测试覆盖

### Pinia 状态管理

项目当前主要包含以下状态模块：

- `auth`：登录弹窗状态、登录态与鉴权判断
- `main-page`：站点列表、当前站点、当前月份、当前查询类型
- `prediction`：模型列表、预测结果、报表开关与展示状态

### composables 业务层

组合式函数将交互流程从组件中抽离，当前核心模块包括：

- `useMainQuery`：主查询页表单状态、条件编辑、分页查询与结果装配
- `useStationDates`：按站点拉取可用日期列表并处理加载态
- `useDataAnalysis`：相关性分析表单、日期范围校验、图表数据管理
- `usePrediction`：预测参数管理、模型类型切换、结果加载
- `useSyncTasks`：同步/清洗待处理日期构建、进度统计与任务执行

### service / api 分层

- `api` 层负责定义 HTTP 请求，不处理复杂业务逻辑
- `service` 层负责统一错误处理、接口结果规范化和前端可用数据结构输出
- `modules` 层负责纯函数计算，例如查询类型转换、预测结果格式化、同步进度计算

### sessionStorage 缓存

`src/lib/storage.js` 对 `sessionStorage` 进行了统一封装，当前缓存内容包括：

- 登录会话信息
- 同步任务涉及的站点列表
- 每个站点待同步日期列表
- 每个站点待清洗日期列表

这样可以在同步/清洗过程中保留前端任务上下文，减少重复查询和状态丢失。

## 接口与运行约定

### 鉴权头约定

当请求配置中包含 `meta.auth = true` 时，`api-client` 会自动从登录会话中读取：

- `name`
- `access_token`

并将其写入请求头。

### 超时重试约定

当请求配置中包含 `meta.retryOnTimeout = true` 且请求因为超时失败时，`api-client` 会自动重试一次，避免偶发网络抖动直接影响页面体验。

### 页面访问约定

- 未登录访问受保护页面时，会被重定向到首页并带上原始目标地址
- 已登录用户访问首页时，会被直接重定向到主工作台页面

## 功能亮点

### 1. 用户登录鉴权与受保护路由

**设计目标**

- 将访客流量与业务操作流量分离，确保核心查询、同步、分析与预测页面仅对已登录用户开放
- 降低每个页面重复编写登录判断逻辑的成本，把鉴权收口到统一入口

**交互与数据流**

- 用户在首页打开登录或注册弹窗提交身份信息
- `auth-service` 调用认证接口，成功后将会话写入 `sessionStorage`
- 路由切换时由全局前置守卫读取 `auth` store 的登录态
- 若目标路由需要鉴权但当前未登录，则跳回首页并保留重定向目标

**关键实现点**

- `src/stores/auth.js` 统一维护弹窗状态、加载态与登录态
- `src/lib/storage.js` 对会话结构做归一化校验，只接受包含 `name` 和 `access_token` 的完整会话
- `src/router/guards.js` 将鉴权判定抽成纯函数，既减少副作用，又能被 `tests/router-guards.test.js` 精准覆盖
- 这种实现将“会话来源、状态存储、路由跳转”三件事解耦，后续若要接入刷新令牌或更复杂权限模型，也能在现有结构上平滑扩展

### 2. 多粒度气象数据查询与复合条件筛选

**设计目标**

- 同时满足“快速看单日变化”和“跨时间段检索异常数据”两类分析需求
- 用统一表单承载多种查询类型，减少页面切换成本

**交互与数据流**

- 用户在主查询页选择站点、气象要素和时间条件
- 查询类型由页面状态决定，表单根据类型切换小时、单日、时间范围或复合条件输入项
- `useMainQuery` 负责表单联动、日期合法性约束、条件范围编辑和分页偏移计算
- `meteo-data-service` 根据查询类型分发到不同接口，并统一回收为表头、表格数据和总数

**关键实现点**

- `src/modules/query/query-helpers.js` 统一处理要素编码排序、分页偏移、日期范围校正、复合条件载荷生成
- `useStationDates` 按选中站点动态拉取有效日期，保证日期选择器只允许选择真实存在的数据区间
- 复合条件查询通过动态生成 `start_xxx / end_xxx` 载荷，避免在组件中硬编码各类气象字段
- 查询结果表头由所选气象要素反向生成，前端展示与查询参数天然保持一致，减少错列和错位风险

### 3. 数据同步、待处理日期缓存与数据清洗流程

**设计目标**

- 为多站点、多日期的数据接入提供可视化、可恢复的批量执行入口
- 在同步和清洗两个阶段之间建立明确的前后依赖，避免重复执行和无效调用

**交互与数据流**

- 前端先连接同步服务，再同步站点目录与远端日期范围
- 对每个站点同时获取本地有效日期、远端最新同步日期、远端是否已有数据
- 根据比较结果推导“待同步日期”和“待清洗日期”，并缓存到 `sessionStorage`
- 执行任务时按站点、按日期逐项推进，同时实时计算百分比进度

**关键实现点**

- `useSyncTasks` 把准备阶段、同步阶段、清洗阶段拆成独立流程，并共享同一套进度统计逻辑
- `src/modules/sync/sync-helpers.js` 提供日期排序、待处理计数和百分比计算等纯函数，降低流程代码复杂度
- `src/lib/storage.js` 以 `sync` 和 `clean` 两类前缀缓存待处理日期，保证两个流程互不污染
- 同步和清洗完成后会重新刷新状态，重新计算未完成任务，确保页面呈现与服务端最新状态保持一致

### 4. 气象要素相关性分析与图表展示

**设计目标**

- 让用户不只看到原始数据，还能快速洞察温度、湿度、风速、风向、降雨、光照、PM2.5、PM10 等要素之间的关系
- 用可交互图表替代纯文本矩阵，提高分析效率

**交互与数据流**

- 用户选择站点、起止日期和至少两个气象要素后发起分析
- `useDataAnalysis` 拉取分析结果并对请求进行并发安全控制，避免旧请求覆盖新结果
- `chart-service` 将接口返回的相关系数矩阵转换为 ECharts 热力图所需数据结构
- 图表组件根据当前选择要素动态更新坐标轴与热力图单元格

**关键实现点**

- `requestId` 机制保证快速切换筛选条件时，只有最后一次请求结果会落到界面上
- 相关性热力图使用 `visualMap` 将系数映射到颜色空间，既适合展示正相关，也适合展示负相关
- 要素标签由常量映射统一提供，避免接口编码与展示名称之间散落转换逻辑
- 与查询模块共用站点有效日期能力，减少重复实现并保证时间约束一致

### 5. 基于模型类型切换的短期/长期预测与报告展示

**设计目标**

- 将预测参数配置、模型切换、结果展示、报告查看组织成完整闭环
- 兼顾短期滚动预测与长期趋势预测两类使用场景

**交互与数据流**

- 用户在预测页选择站点、日期和预测方案
- `usePrediction` 根据方案切换模型类型，并结合有效日期自动修正可选时间
- `meteo-data-service` 发起预测接口调用并返回标准化结果
- `prediction-helpers` 将二维预测数组加工为表格可直接消费的行数据，结果页按列头渲染
- 模型报告接口返回后，可与预测结果一起用于解释当前模型表现

**关键实现点**

- 预测方案与模型类型映射被抽到 `src/modules/prediction/prediction-helpers.js`，避免在组件里写死条件分支
- 短期预测按小时生成未来时间轴，长期预测按自然日生成未来日期轴，确保结果具有可读的时间语义
- `src/stores/prediction.js` 用于统一保存预测结果和模型展示状态，便于头部配置区与结果区共享数据
- 预测结果表格通过“表头常量 + 二维数组转对象数组”的方式渲染，既适配多指标输出，也让后续图表化扩展更直接
